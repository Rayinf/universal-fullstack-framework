# AI 协作开发指南

本文不是泛泛讲“AI 很有帮助”，而是专门讲：  
同事在这个仓库里应该怎样使用 AI，才能持续接手开发，而不是把仓库越改越乱。

## 1. 先讲清楚：在这个项目里，AI 应该扮演什么角色

在这个仓库里，AI 更适合扮演下面 4 种角色：

1. 项目基座的上下文整理者
2. 标准脚手架和 registry 工作流的执行者
3. 路由/菜单/权限/后端菜单树联动问题的排查者
4. 代码修改后的验证与回归助手

AI 不应该扮演的角色是：

1. 绕开项目约束自由发挥
2. 不看当前代码就凭旧印象改仓库
3. 为了“快”而跳过脚手架、registry 和 baseline 验证

一句话理解：

> 在这个仓库里，AI 不是“随便帮你写几段代码”，而是“按项目约束执行开发流程的工程助手”。

## 2. 同事第一次使用 AI 接手这个仓库时，应该先做什么

建议固定使用下面这个顺序。

### 2.1 先让 AI 读项目约束

至少让 AI 先读取这些文件：

1. `AGENTS.md`
2. `CLAUDE.md`
3. `docs/training/framework-baseline-training.md`
4. `docs/training/skills-training.md`
5. `docs/training/scaffold-registry-verify-training.md`

如果任务是新增模块，再补读：

1. `docs/fullstack-module-template.md`
2. `docs/frontend-module-template.md`
3. `backend/docs/backend-module-template.md`

### 2.2 再让 AI 看当前代码，而不是只看旧文档

需要强调给同事：

1. 当前代码状态要以实际文件为准
2. 不要让 AI 只根据仓库旧 README 推断结构
3. 尤其要让 AI 检查：
   - `src/config/frameworkConfig.ts`
   - `src/router/index.ts`
   - `src/config/menuConfig.ts`
   - `src/layouts/MainLayout.vue`
   - `backend/app/modules/system_admin/menu.py`
   - `scripts/verify_framework_baseline.sh`

### 2.3 然后再给 AI 明确任务边界

至少要告诉 AI：

1. 任务是前端、后端还是全栈
2. 目标业务模块是什么
3. 预期页面或接口是什么
4. 需要复用哪类脚手架
5. 哪些验证命令必须跑

## 3. 给 AI 的上下文，至少要包含什么

在这个仓库里，一个能让 AI 真正干活的任务描述，至少要有下面这些信息。

### 3.1 业务目标

例如：

1. 新增质检报告模块
2. 修复销售合同页的菜单权限问题
3. 重构某后端模块为标准分层结构
4. 给某模块补一页 CRUD 和对应后端接口

### 3.2 技术边界

例如：

1. 只改前端
2. 只改后端
3. 前后端一起改
4. 只做代码审查，不要改代码

### 3.3 模块接入信息

如果是新增模块，尽量给全这些字段：

1. `module_name`
2. 中文 `tag`
3. `menu_parent`
4. `route_path`
5. `route_name`
6. `function_code`
7. `api_base_path`
8. `table_name`
9. 是否 `with-store`

### 3.4 验收标准

至少告诉 AI：

1. 页面是否需要可访问
2. 菜单是否需要可见
3. 权限码是否必须接通
4. 是否要跑 `bash scripts/verify_framework_baseline.sh`
5. 是否要补 smoke 手工验证路径

## 4. 在这个仓库里，推荐让 AI 遵循的固定工作流

无论任务大小，建议都按这个顺序做。

### 4.1 先理解任务

AI 应该先确认：

1. 任务目标是什么
2. 影响前端还是后端还是两边
3. 需要接入哪些模块路径
4. 会不会影响权限、菜单、registry、seed

### 4.2 再读代码和文档

优先读：

1. 当前任务相关代码
2. 当前仓库相关模板文档
3. 对应 skill 的 `SKILL.md`

### 4.3 再选择脚手架还是手改

默认原则：

1. 标准新增模块，优先脚手架
2. 小范围修复，优先局部手改
3. 跨前后端标准扩展，优先全栈脚手架

### 4.4 再实施修改

实施时要遵守：

1. 主文件尽量保持稳定
2. 优先写 registry
3. `meta.functionCode` 与后端 `permission` 必须一致
4. 不要在前端私自写死 mock 代替后端 seed

### 4.5 最后做验证

至少要做下面三类检查：

1. 代码是否通过编译/类型检查
2. registry 是否正确接线
3. 功能链路是否能通过 smoke 或 baseline 验证

## 5. 哪些任务最适合直接交给 AI

### 5.1 新增标准 CRUD 模块

非常适合。

前提是你把这些信息告诉 AI：

1. 模块中文名
2. 接口前缀
3. 表名
4. 菜单挂载位置
5. 权限码
6. 核心字段

### 5.2 排查路由/菜单/权限联动问题

非常适合。

这类任务最适合让 AI 同时检查：

1. `src/router/index.ts`
2. `src/config/menuConfig.ts`
3. `src/config/frameworkConfig.ts`
4. `src/layouts/MainLayout.vue`
5. `backend/app/modules/system_admin/menu.py`
6. `src/stores/menuStore.ts`

### 5.3 做代码审查

也很适合。  
尤其适合让 AI 重点看：

1. 是否破坏基座结构
2. 是否遗漏 registry
3. 是否遗漏 `functionCode / permission`
4. 是否跳过验证

### 5.4 整理文档和培训材料

也很适合。  
因为这类任务可以充分利用 AI 的归纳和结构化能力。

## 6. 哪些事 AI 可以直接做，哪些要先确认，哪些必须人工拍板

你这个仓库里最容易讲模糊的，不是“AI 能不能做”，而是“做到什么程度可以直接做”。

正确的划分应该是三层。

### 6.1 第一层：skill 或仓库规范已经明确定义，AI 应直接执行

只要这些事项已经在 skill、AGENTS、模板文档或当前仓库代码里明确下来，AI 就不应该再反复问人，而应该直接按标准执行。

典型例子：

1. 标准新增模块优先使用脚手架，而不是手工复制旧模块
2. 新模块优先写入 registry，而不是持续堆主文件
3. 前端 `meta.functionCode` 与后端 `permission` 必须一致
4. 所有 ID 字段保持 `string` 语义
5. API 成功按 `code = 0` 或 `200` 处理
6. 前端优先使用 `@/` 路径别名
7. 异步逻辑保留完整错误处理
8. 涉及共享基础设施时优先跑 `bash scripts/verify_framework_baseline.sh`

这类事情，本质上已经不是“开放设计题”，而是“按既定规则执行”。

### 6.2 第二层：仓库给了方向，但具体值不完全明确，AI 应先提案，再请人确认

这类事情可以先让 AI 提出建议方案，但不要默认由 AI 单方面拍板。

典型例子：

1. 新模块应该挂在 `system`、`sales` 还是 `production`
2. 某个新功能的 `functionCode` 应该怎么命名，若当前没有现成编号
3. 菜单标题、菜单图标用什么更贴合业务语义
4. 某个模块应该沿用现有页面目录，还是顺手做目录迁移
5. 只知道大致需求时，查询条件、表格列、表单字段应如何取舍

这类事项的推荐做法是：

1. AI 先给 1 个推荐方案
2. 同时说明理由和影响范围
3. 再由人快速确认

### 6.3 第三层：会改变业务语义或基座边界的事，必须人工拍板

这类事情不适合让 AI 自己决定。

典型例子：

1. 新模块的真实业务字段定义和业务流程
2. 数据表结构和真实 SQL 设计的最终业务语义
3. 是否引入新的一级业务域或新的路由前缀
4. 是否允许改动认证、菜单树、权限语义这些基座主干规则
5. 是否接受某种 breaking change 或兼容性策略

这类事情不是 AI 做不到，而是它们本质上属于“产品/架构决策”，需要有人承担最终责任。

### 6.4 最适合教给同事的判断方法

可以让同事先问自己一句话：

“这件事在 skill、仓库规范或现有代码里，是不是已经有明确答案？”

如果答案是：

1. `是`
   让 AI 直接执行
2. `大致有方向，但值不唯一`
   让 AI 先提案，人确认
3. `没有明确标准，而且会改变业务或基座边界`
   必须人工拍板

一句话总结：

> 在这个项目里，不是“重要的都得人工决定”，而是“已经被 skill 和仓库规范固化的，AI 就应直接执行；只有超出既有规范边界的，才需要人工拍板”。

## 7. 给 AI 下任务时，最常见的错误

### 7.1 任务描述过于口语化

错误示例：

- “帮我把这个页面弄一下”
- “这个模块补完整”

这类描述对 AI 来说缺少：

1. 修改范围
2. 接入位置
3. 验收标准

### 7.2 只说结果，不说约束

错误示例：

- “给我做一个质检页面”

但没有告诉 AI：

1. 用哪个脚手架
2. 菜单挂哪里
3. 权限码是什么
4. 是否要跑 baseline

### 7.3 不让 AI 先看当前代码

这会导致 AI 按旧认知做判断。  
在这个仓库里尤其危险，因为旧版 README 和当前实际启用模块并不完全一致。

### 7.4 让 AI 只写代码，不做验证

在这个仓库里，代码写完不是结束。  
必须至少要求 AI 给出：

1. 改了哪些文件
2. 跑了哪些验证
3. 还有哪些风险

## 8. 推荐同事给 AI 的输出格式要求

每次都建议在 prompt 里明确要求 AI 最后输出：

1. 改了哪些文件
2. 为什么这样改
3. 运行了哪些命令
4. 哪些验证通过了
5. 还有哪些风险或未做项

这样有两个好处：

1. 便于人快速审阅
2. 便于后续继续接力

## 9. 多 agent 并行时怎么拆最稳

如果同事使用的 AI 工具有多 agent 或并行子任务能力，建议按“最小冲突写域”拆分。

推荐拆法：

1. 一个 agent 只负责项目现状和联动点梳理
2. 一个 agent 只负责前端页面和前端 registry
3. 一个 agent 只负责后端模块和后端 registry
4. 一个 agent 只负责测试、验证和 review
5. 一个 agent 负责最后整合

不推荐的拆法：

1. 多个 agent 同时改同一个主文件
2. 多个 agent 同时改同一个模块目录
3. 让一个 agent 改代码，另一个 agent 在不知道上下文的情况下重写同一部分

## 10. AI 输出拿回来后，人应该怎么审

建议用下面这张清单快速过一遍：

1. 有没有绕开脚手架和 registry
2. 有没有直接把主文件改成更乱
3. `functionCode` 和后端 `permission` 是否一致
4. `frameworkConfig.ts` 是否允许对应前缀
5. 后端菜单树和角色授权链路是否仍成立
6. 是否跑了类型检查、测试或 baseline
7. 是否留下了占位字段、占位错误文案、占位 SQL

## 11. 推荐给同事的“首次启动 AI 会话”动作

第一次接手时，建议直接让 AI 做下面这件事：

1. 读取 `AGENTS.md` 和 `CLAUDE.md`
2. 读取 `docs/training/` 下的培训材料
3. 读取 `src/router/index.ts`、`src/config/menuConfig.ts`、`backend/app/modules/system_admin/menu.py`
4. 总结当前启用模块、脚手架机制、registry 机制和 baseline 验证链路
5. 再开始当前具体任务

这一步的价值是：

让 AI 先变成“了解当前仓库实际结构的协作者”，再去动代码。

## 12. 一句话总结

> 在这个项目里，把 AI 用好的关键，不是“会不会写 prompt”，而是“能不能让 AI 先进入正确的工程上下文，再按基座、skill、脚手架、registry、验证闭环这条路径工作”。
