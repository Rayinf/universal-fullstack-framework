# Skills 搭建思路与使用说明

本文面向第一次接触 skill 的同事，目标是讲清楚：  
为什么这个项目需要 skill、skill 的结构是什么、它怎样和项目基座配合、实际使用时该怎么做。

## 1. 先用最容易懂的话解释 skill

可以直接对新人这样说：

> skill 不是普通文档，也不是知识库，而是一套可触发的标准工作流。

它的作用不是“告诉 agent 一堆常识”，而是把团队已经验证过的做法、约束、工具和注意事项，打包成一套能复用的操作说明。

## 2. 为什么这个项目需要 skill

如果没有 skill，agent 每次都会面临同样的问题：

1. 要自己重新猜测项目结构
2. 要自己判断哪些文件能改、哪些不该改
3. 要自己猜新增模块时该改路由还是菜单还是后端
4. 容易遗漏多文件联动点
5. 容易用“看起来能跑”的方法破坏基座稳定性

而这个仓库恰恰是一个：

- 前后端联动很强
- 多个大文件需要保持稳定
- 新模块要靠脚手架和 registry 扩展
- 改完必须过基座验证链路

的工程体系。

所以 skill 的价值就在于：

1. 把通用大模型变成遵守项目约束的专项 agent
2. 把高频、易错、多文件联动的任务固化为统一流程
3. 把“先做什么、后做什么、不能漏什么”固定下来
4. 避免每次都从零推理

## 3. skill 在这个仓库里解决的核心问题

最关键的一类 skill 是：`universal-fullstack-framework`

它的作用可以概括成三句话：

1. 先保护基座，避免把系统再次改回“大而乱”的业务堆栈
2. 再基于脚手架和 registry 扩展模块
3. 始终把路由、菜单、权限、后端接口、验证闭环作为一个整体来处理

另一类关键 skill 是：`skill-creator`

它解决的问题是：

1. 如何把团队经验沉淀成 skill
2. skill 应该写什么，不该写什么
3. skill 的信息该如何分层加载，避免上下文膨胀

## 4. 一个 skill 的典型结构

根据 `skill-creator` 的设计，一个 skill 通常由以下部分组成：

### 4.1 `SKILL.md`

这是必需文件。

它包含两部分：

1. YAML frontmatter
   - 最重要的是 `name`
   - 以及 `description`
2. 正文
   - 写清流程、规则、何时读取 reference、何时运行 script

需要给新人强调：

`name + description` 决定 skill 会不会被触发。  
正文只有在 skill 被触发后才会真正被读取。

### 4.2 `agents/openai.yaml`

这是推荐但非必须的 UI 元数据文件。

主要作用：

1. 给 skill 列表和 UI 展示用
2. 提供显示名、短描述、默认提示词

### 4.3 `scripts/`

这里放的是高频、易错、需要确定性执行的脚本。

适合放进去的内容包括：

1. 生成器
2. 转换器
3. 一键验证脚本
4. 需要固定参数与输出格式的命令流程

一句话解释：

`scripts/` 负责把“人容易写错”的动作收敛成稳定工具。

### 4.4 `references/`

这里放详细资料，按需读取。

适合放的内容包括：

1. API 约定
2. 目录规范
3. 业务规则
4. 模板参数说明
5. 长文档参考资料

一句话解释：

`references/` 负责补细节，但不该一上来全塞进上下文。

### 4.5 `assets/`

这里放最终产物要用到的资源，但不一定需要完整读入上下文。

比如：

1. 模板文件
2. 图片
3. 字体
4. 样例素材

## 5. 为什么 skill 要强调 progressive disclosure

这是理解 skill 的关键。

### 5.1 什么叫 progressive disclosure

可以理解为三级加载：

1. 元数据常驻
2. `SKILL.md` 在触发时加载
3. `references/` 和 `scripts/` 只有在需要时才再读取或执行

### 5.2 为什么这样设计

因为上下文窗口不是无限的。

如果把所有知识一次性塞给 agent，会带来三个问题：

1. 关键信息被淹没
2. 推理成本变高
3. 任务结果反而更不稳定

因此，skill 的设计原则不是“写得越全越好”，而是：

1. 只保留模型默认不知道的项目特例
2. 先给核心流程
3. 细节按需展开

### 5.3 在这个仓库里怎么体现

这个仓库里的做法非常典型：

1. `universal-fullstack-framework` 的 `SKILL.md` 主要写：
   - 基座优先
   - 合同先行
   - 优先复用 repo-local scaffolds
   - route/menu/permission/layout 联动
   - 验证闭环
2. 更具体的参数、输出目录、命令示例，放在仓库文档里：
   - `docs/frontend-module-template.md`
   - `docs/fullstack-module-template.md`
   - `backend/docs/backend-module-template.md`
   - `docs/ci-baseline.md`

也就是说：

- skill 负责“先做什么、后做什么、哪些不能漏”
- 仓库文档负责“具体命令、具体路径、具体模板”

## 6. skill 和项目基座是什么关系

可以把二者关系讲成一张逻辑图：

1. 项目基座提供稳定结构
2. skill 提供统一做法
3. 脚手架提供标准化落地
4. registry 提供低冲突扩展入口
5. verify/test 提供回归闸门

培训时可以直接说：

> 基座负责“能不能承载”，skill 负责“应该怎么做”，脚手架负责“如何快速落地”，验证脚本负责“有没有把基座搞坏”。

## 7. AGENTS 规则对实际工作的影响

这个仓库的 AGENTS 约束意味着：

1. 如果任务明显匹配某个 skill，就必须启用它
2. 先读 `SKILL.md`，不要一上来全仓乱扫
3. 如果 skill 提供 `scripts/` 或模板，优先复用
4. 只按需读 `references/`
5. 多 skill 同时适用时，要选最小集合并说明顺序

这几条非常重要，因为它们决定了 agent 在这个仓库里不是“随便做”，而是“按项目工作流做”。

## 8. 这个项目里建议优先让新人认识的两个 skill

### 8.1 `universal-fullstack-framework`

适合用于：

1. 讲项目基座
2. 讲模块扩展
3. 讲前后端联动排错
4. 讲为什么优先走 scaffold 和 registry

### 8.2 `skill-creator`

适合用于：

1. 讲 skill 的设计方法
2. 讲 skill 怎么拆内容层次
3. 讲为什么不要把 skill 写成冗长 README
4. 讲如何沉淀团队经验

## 9. 给新同事的实际使用步骤

### 9.1 第一步：先判断这是不是 skill 任务

如果任务涉及下列内容，应优先想到 skill：

1. 基座整理
2. 模块新增
3. 前后端联动排错
4. 路由菜单权限问题
5. 脚手架生成
6. skill 自身设计或维护

### 9.2 第二步：先读 `SKILL.md`

不要一开始就把所有文档、所有源码都装进脑子里。  
先通过 `SKILL.md` 明确：

1. 这次任务的推荐路径是什么
2. 应优先读取哪些文件
3. 哪些脚本已经可以复用
4. 结束前要做哪些验证

### 9.3 第三步：按需读取 reference 或执行 script

比如本仓库里：

1. 要新增全栈模块，就进一步看 `docs/fullstack-module-template.md`
2. 只补前端模块，就看 `docs/frontend-module-template.md`
3. 只补后端模块，就看 `backend/docs/backend-module-template.md`
4. 要跑统一回归，就看 `docs/ci-baseline.md`

### 9.4 第四步：沿着基座约束落地

在这个项目里，实际落地时最重要的约束包括：

1. 大主文件尽量保持稳定
2. 新模块优先写 registry
3. `meta.functionCode` 与后端 `permission` 必须一致
4. 新增页面不能只做前端，不管后端菜单树
5. 改完必须跑验证链路

### 9.5 第五步：把结果当成“稳定交付”，不是“文件生成”

这是培训时必须反复强调的观念：

skill 的目标不是多生成几个文件，  
而是让修改过程具备：

1. 一致性
2. 可复用性
3. 可验证性
4. 可维护性

## 10. 新人最容易误解的几点

### 10.1 skill 不是知识库

知识库偏向“把东西放在那里供查找”。  
skill 偏向“触发后指导 agent 如何做事”。

### 10.2 skill 不是替代代码结构

skill 不能代替路由、菜单、后端模块这些工程结构。  
它只是规定如何正确使用这些结构。

### 10.3 skill 不是越长越好

过长的 skill 会让上下文污染严重。  
真正好的 skill 应该：

1. 主流程短
2. 关键约束清楚
3. 细节下沉到 reference
4. 确定性动作收进 script

### 10.4 skill 不能替代验证

就算完全按 skill 做，也不能省略类型检查、单测、smoke、构建检查。

## 11. 给受训同事的一句话总结

> skill 是这个项目的“标准操作说明书”，它让 agent 和开发者都不必每次从零摸索，而是围绕稳定基座、脚手架、registry 和验证闭环协同工作。
