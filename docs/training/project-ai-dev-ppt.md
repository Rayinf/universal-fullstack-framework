## Slide 3 这套项目现在用什么技术栈

### 核心信息

- 前端这一侧
  - `Vue 3 + TypeScript + Vite`
  - `Pinia + Vue Router`
  - `Element Plus + Axios`
  - 主要负责页面、状态、路由和请求
- 后端这一侧
  - `FastAPI + PostgreSQL`
  - 登录权限、JWT 和演示数据
  - `router / service / repository` 分层
  - 主要负责菜单、权限、接口和数据
- 开发和生成
  - 前端 / 后端 / 全栈三类脚手架
  - 用来先把模块骨架起出来
- 验证和交付
  - Python 语法、单测、`type-check`、构建、`baseline`
  - 用来统一验收，不靠感觉说“做完了”

### 一句话理解

前端做页面，后端管权限和数据，脚手架起步，`baseline` 验收。

### 页面目的

先知道前端、后端、验证各自用什么，后面看代码时才不容易发懵。

## Slide 6 系统地图：前端、后端、验证各管什么

### 核心信息

- 前端负责你看到和点到的东西：
  - 页面、表格、表单
  - 页面跳转和菜单显示
  - 登录后的页面状态
  - 这个页面前端这边让不让进
- 后端负责系统认不认、给不给数据：
  - 菜单树和权限码
  - 角色到底能看哪些菜单
  - 业务接口和模块路由
  - 初始化演示数据
- 验证负责最后兜底：
  - Python 能不能正常跑
  - 后端公共能力有没有被改坏
  - 接口能不能正常通
  - 前端类型和构建能不能过

### 一句话理解

一个功能打不开，常见不是页面写错，而是前后端和权限这条线没有对上。

### 页面目的

先搞清楚系统是怎么分工的，再去理解为什么这里不能只盯着一个页面改。

## Slide 7 这个基座其实分成 5 层

### 核心信息

- 页面壳和导航层
  - 统一布局、侧边菜单、顶部栏、移动端容器
  - 新页面不用从头搭外框
- 登录和权限层
  - 登录态、角色、菜单权限、页面拦截
  - 很多“能不能进页面”的判断已经在基座里
- 前端公共能力层
  - `request`、`store`、通用表格表单、CRUD 写法
  - 做页面时通常只补业务字段和交互
- 后端公共能力层
  - 应用启动、数据库、菜单树、seed、模块化 `router / service / repository`
  - 这些骨架已经在
- 生成和验证层
  - `scaffold`、`registry`、`baseline`
  - 新模块按固定入口生成，改完按固定方式验

### 页面目的

后面新模块能接得快，不是因为 AI 会“魔法生成”，而是这些公共层已经先搭好了。

## Slide 10 这套基座里，哪些规则已经固定下来了

### 核心信息

- 红线 1：ID 和成功码别乱
  - ID 默认用 `string`
  - 接口成功按 `code = 0` 或 `200` 处理
  - 否则前后端一对接就容易出怪问题
- 红线 2：权限四件套要一起改
  - `route`、`menu`、`layout`、`permission` 要一起改
  - 前端 `functionCode` 要和后端 `permission` 对齐
  - 少一处就可能出现“菜单有了，但页面进不去”
- 红线 3：启用边界看 `frameworkConfig.ts`
  - 哪些业务域现在能用，以它为准
  - 没启用的前缀不要直接当正式入口用
- 红线 4：新增模块先 `scaffold`，改完跑 `baseline`
  - seed 放 `init_db`
  - 不要手工复制老模块
  - 共享改动要跑统一验证

### 页面目的

这些不是“建议”，而是后面继续开发时默认要遵守的公共约定。

## Slide 14 用这个基座继续开发，常见就是 4 类事情

### 常见任务类型

1. 新增完整业务模块
2. 给现有模块补页面
3. 给现有模块补后端能力
4. 修联动问题或做重构

### 每类任务怎么理解

- 给现有模块补页面
  - 后端已经有了
  - 只补列表、表单、详情或交互
- 给现有模块补后端能力
  - 前端已经有了
  - 只补接口、权限、演示数据或后端结构
- 修联动问题或做重构
  - 重点不是某个页面
  - 而是 `route / menu / permission / registry / baseline` 这一串

### 页面目的

你后面遇到的事，大多都能归到这 4 类。先分对类，AI 才好接手。

## Slide 15 标准扩展路径：从需求到交付

### 标准路径

1. 先定挂到哪
2. 再选怎么做
3. 先把骨架搭出来
4. 把该接的都接上
5. 再补真实业务
6. 最后统一验证并交付说明

### 关键动作

- 先判断挂到 `system`、`sales` 还是 `production`
- 再决定是 `fullstack`、`frontend`、`backend` 还是直接在旧模块上改
- 先把目录、页面、接口和基础接线点生出来
- 再把 `route / menu / permission / registry / seed / 授权` 这条线接齐
- 最后把占位字段、SQL、文案和真实业务逻辑换成正式内容

### 页面目的

别急着开写。先把要挂哪里、怎么接进去、最后怎么验清楚。

## Slide 16 真实扩展示例：新增一个质检报告模块

### 示例背景

- 场景：新增一个 `quality_report` 模块
- 要求：菜单能出现、页面能打开、后端有接口、本地有演示数据、改完能过统一验证
- 关键判断：虽然叫“质检”，但现在先挂 `production`，不另开 `quality` 一级模块

### 6 步走法

1. 先决定放哪块
   - 先挂 `production`
2. 选择 `fullstack scaffold`
   - 因为页面、接口、菜单、权限都要一起补
3. 把参数先定清
   - 模块名、挂载位置、路由、功能码、表名这些先定好
4. 先出骨架
   - 先生成 `view / api / type / store / backend module`
5. 补齐接线
   - 把 `route / menu / permission / registry / seed / 授权` 补上
6. 换成真实业务再验
   - 把占位字段和 SQL 换掉，再跑 `baseline`

### 给 AI 的 5 个必填项

- 模块名：`quality_report`
- 业务域：`production`
- 路由路径：`/production/quality-report`
- 功能码：`production-quality-report`
- 权限码：`SRS-FUNC-QUALITY-REPORT`

### 页面目的

拿一个最像真实工作的例子，把整条扩展路径讲明白。

## Slide 21 先认识 Skill：它到底是什么

### 核心信息

- 这个词别想复杂
- 你可以把 Skill 理解成 AI 的固定工作办法
- 它更像一份工作说明书：
  - 先读什么
  - 按什么顺序做
  - 哪些步骤不能跳过
- 它帮你省掉什么：
  - 不用每次都从头给 AI 讲规矩
  - 不同同事接手时，也能让 AI 按同一套方法做事
- 里面一般会写什么：
  - 什么时候用
  - 先看哪些文件
  - 推荐步骤
  - 默认验证
  - 常见坑位和不能跳过的护栏
- 它不负责什么：
  - 不是魔法插件
  - 不是替你决定真实业务
  - 不是绕开项目现有流程的捷径

### 一句话理解

`skill 的价值，不是让 AI 更会聊天，而是先让它按对的方法干活。`

### 页面目的

## Slide 22 这个项目专用的 Skill

### 核心信息

- skill 名称：
  - `universal-fullstack-framework`
- 什么时候用：
  - 新增模块
  - 整理或重构基座
  - 排查前后端联动问题
  - 把 mock 或远程依赖换成本地实现
  - 共享链路回归加固
- 它默认怎么干：
  - 先保护公共基座别改乱
  - 先把路由、菜单、接口对齐
  - 优先用脚手架和注册点
  - 优先接本地 `FastAPI + PostgreSQL`
- 它默认盯什么：
  - `functionCode` 要对上 `permission`
  - `route / menu / layout` 要一起动
  - 新增模块先做最小可用
  - 共享改动优先跑 `baseline`

### 一句话理解

它不是替你决定业务，而是把“怎么沿着现有基座继续做”先写成 AI 必须遵守的流程。

### 页面目的

让同事知道：这个项目不是“随便和 AI 说一句需求”就开始写，而是优先走项目专用 Skill。

## Slide 23 这个专属 Skill 用了以后，AI 会按什么顺序做

### 核心信息

1. 先保住基座
   - 先确认哪些是公共底座，尽量别为了新需求把主干改乱
2. 先对齐前后端合同
   - 先看 `route`、`menu`、API、字段、返回格式是不是一一对应
3. 优先接本地后端
   - 优先走 `FastAPI + PostgreSQL + seed`
   - 不要先靠 mock 顶着
4. 新增模块先用脚手架
   - 前后端都要改时优先 `fullstack scaffold`
   - 其次再拆 `frontend / backend`
5. 最后统一验证
   - 共享改动优先跑 `baseline`
   - 再看 smoke 和关键流程有没有通过

### 一句话理解

所以用了这个 skill 以后，AI 正常不会一上来复制旧模块，也不会跳过验证直接说“已完成”。

### 页面目的

把“这个 skill 实际会怎样约束 AI 的工作顺序”讲清楚。

## Slide 24 这个专属 Skill 会默认盯哪些风险和交付项

### 核心信息

- 联动风险
  - 新增或移动模块时，`route / menu / permission / layout` 这些要一起检查
  - 少一个都可能出问题
- 数据和演示风险
  - seed 不是随便塞几条数据
  - 而是要能支撑一整条业务链跑起来
  - 不要只在前端写假数据
- 环境和依赖风险
  - 像 `python-multipart`、端口占用、虚拟环境装错依赖
  - 这些它都会当成默认检查点
- 交付时要说清什么
  - 改了哪些文件
  - 解决了什么问题
  - 跑了哪些验证
  - 结果如何
  - 还剩哪些风险

### 一句话理解

你可以把这页理解成：这个 skill 不只教 AI 怎么写代码，还会逼它别漏检查、别漏说明。

### 页面目的

把“这个 skill 除了指导步骤，还会额外帮你兜住哪些坑”讲清楚。

## Slide 25 这 4 个词分别干什么

### 核心信息

- Skill
  - 定做法
  - 告诉 AI 先看什么、先做什么、哪些步骤不能跳过
- Scaffold
  - 出骨架
  - 先生成目录、页面、接口和基础接线点

### 页面目的

三者关系其实很简单：skill 管做法，scaffold 出骨架，registry 负责接进去。

## Slide 26 把这份 Skill 翻成人话，可以怎么理解

### 核心信息

- 它先要求：先把基座稳住
  - 先保住认证、系统管理、权限、日志和一条基础 CRUD
  - 不要一上来就为了新需求把主干改乱
- 它再要求：先把前后端对齐
  - 先把前端 API 和后端接口一一对应
  - 缺哪个补哪个，不要两边各猜各的
- 它默认会把 remote / mock 往本地真实链路收
  - 优先接本地 `FastAPI + PostgreSQL + seed`
  - 不要长期靠 mock 顶着
- 它要求统一开发写法
  - 前端走统一 `request / store / CRUD` 模式
  - 后端走模块化分层和注册机制
- 它最后强调：每次都要小步扩展、小步验证
  - 先做一段最小可用
  - 做完就验，再继续往下做

### 一句话理解

`翻成人话：先把底子稳住，再把前后端对齐，然后沿着本地真实链路一点点往外扩。`

### 页面目的

把 skill 原文从“工程说明书”翻成同事更容易听懂的中文版本。

## Slide 27 这个 Skill 会用到哪些主脚本

### 核心信息

- `scripts/scaffold_fullstack_module.py`
  - 同时生成前后端模块骨架
  - 是标准新增模块时的默认入口
- `backend/scripts/scaffold_backend_module.py`
  - 只生成后端模块骨架
  - 会产出 `router / routes / services / repositories`
  - 还会把新模块写进后端路由注册表
- `scripts/scaffold_frontend_module.py`
  - 只生成前端模块骨架
  - 会产出 `types / api / view / store`
  - 还会把路由、菜单和后端菜单注册点补进去
- `scripts/verify_framework_baseline.sh`
  - 是统一的基座验收入口
  - 会把编译、单测、HTTP smoke、`type-check`、构建串起来

### 一句话理解

`可以把它们记成两类：前面三个负责“起骨架”，最后一个负责“验基座”。`

### 页面目的

让同事知道：这个 skill 不是只会“提要求”，它背后其实已经对应好了项目里的具体脚本入口。

## Slide 28 baseline 里每个校验脚本分别在查什么

### 核心信息

- `backend/main.py`
  - 先做主入口语法检查
  - 先保证后端主程序至少能正常加载
- `backend/tests/test_backend_scaffold.py`
  - 检查后端脚手架能不能正确生成模块文件和路由注册
- `backend/tests/test_frontend_scaffold.py`
  - 检查前端脚手架能不能正确生成页面文件、路由和菜单注册
- `backend/tests/test_fullstack_scaffold.py`
  - 检查全栈脚手架能不能把前后端一起接通
- `backend/tests/test_menu_registry.py`
  - 检查后端菜单注册表能不能正确合并进菜单树
- `backend/tests/test_router_registry.py`
  - 检查后端路由注册表能不能把新模块动态挂进应用
- `backend/tests/test_http_smoke.py`
  - 会起本地服务，走登录、菜单、刷新 token 和基础接口 smoke
- `backend/tests/test_modular_routes.py`
  - 检查模块化 router 组合后还能正常工作

### 一句话理解

`所以 baseline 不是“随便跑个脚本”，而是在查：骨架有没有坏、注册有没有坏、接口有没有通、前端能不能真交付。`

### 页面目的

把 baseline 里真正跑到的检查项讲清楚，避免同事只知道“要跑”，不知道“它到底在帮我们兜什么”。

## Slide 29 新增模块时，先判断走哪条入口

### 三类入口

- `backend scaffold`
- `frontend scaffold`
- `fullstack scaffold`

### 默认原则

- 先分清这次只改前端、只改后端，还是两边都改
- 大多数标准新增功能，默认先选 `fullstack scaffold`
- 不要上来就复制旧模块

### 页面目的

入口选对，后面的开发会顺很多；入口选错，后面补救会很麻烦。

## Slide 30 当前边界与不要误判的点

### 常见误判

1. 看到 `sales / production`，不等于完整 MES 做完
   - 它们是在演示这套基座怎么扩，不代表所有真实业务规则都已经定好了
2. 菜单能看见，不等于权限一定正确
   - 还要继续看 `functionCode`、`permission`、`allowedMenuIds` 和页面拦截是不是都对上
3. 脚手架生成完，不等于模块可以交付
   - 生成出来的只是最小骨架；占位字段、文案、SQL、seed 都还要换成正式业务内容
4. 仓库里有 mock 文件，不等于当前默认走 mock
   - 现在培训和联调默认应该走本地 `FastAPI + PostgreSQL + seed`
5. 拿不准当前边界时，不要只凭感觉
   - 先看 `frameworkConfig.ts`、`router`、`menu.py` 和 `verify` 结果

### 页面目的

这页专门提醒：哪些看起来像“已经好了”，其实还没有。

## Slide 31 代码写完，不等于任务完成

### baseline 核心检查

1. Python 语法和关键脚本
   - 不过：一运行就会报 import 或脚本错误
2. 后端公共能力回归
   - 不过：脚手架、注册点和模块路由可能被带坏
3. 接口能不能正常通
   - 不过：登录、菜单、refresh 和 CRUD 这条线会断
4. 前端类型是否正常
   - 不过：Vue / TS 类型会被这次改动带坏
5. 前端构建是否正常
   - 不过：页面能开，发布却过不了 build

### 统一入口

`bash scripts/verify_framework_baseline.sh`

### 页面目的

代码能运行，不等于这次任务真的可以交。

## Slide 32 和 AI 协作时的标准流程

### 5 步流程

1. 先说清任务
   - 说清这次要做什么、不要改什么、验收到什么程度
2. 先把规则喂给 AI
   - 先读 `AGENTS`、training 文档和对应 skill
3. 再让 AI 读现状
   - 先看当前启用边界、路由、菜单和关键入口
4. 再开始改
   - 标准新增模块优先走 `scaffold`，局部问题再手改
5. 最后要求验证和说明
   - 说清修改文件、验证结果和剩余风险

### 页面目的

这页不是讲模块怎么接，而是讲你和 AI 怎么配合不跑偏。

## Slide 33 常见问题从哪里开始查

### 典型问题

- 菜单不显示
- 页面被重定向
- 模块生成后不完整
- 验证失败

### 默认排查入口

- 菜单不显示
  - 先看这个前缀有没有启用，再看后端菜单树和 registry 有没有接进去
- 页面被重定向
  - 先看 `functionCode`、`permission`、`allowedMenuIds` 这些权限相关的值有没有对上
- 模块生成后不完整
  - 先看脚手架是不是走对了，再把占位字段、文案和 SQL 换成真的
- 验证失败
  - 先看卡在哪一步：Python、后端测试、接口、`type-check` 还是 `build`

### 页面目的

先按固定顺序查，别一出问题就盯着页面代码猛改。
